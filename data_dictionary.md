# Data Dictionary

This project uses deterministic synthetic clickstream data. The data is generated by `scripts/generate_synthetic_data.py` with seed `42`.

## Grain

Each row is one product event. Multiple events belong to one session through `session_id`.

## Fields

| Column | Type | Description |
| --- | --- | --- |
| `event_id` | string | Unique event identifier. |
| `user_id` | integer | Synthetic user identifier. |
| `session_id` | string | Synthetic session identifier. |
| `event_type` | string | One of `app_open`, `home_impression`, `video_play`, `video_complete`, `video_stop`, `app_close`. |
| `timestamp` | ISO datetime | Event timestamp in UTC. |
| `region` | string | User region. |
| `device` | string | User device category. |
| `app_version` | string | Client app version. |
| `model_version` | string | Recommendation model version served to the session. |
| `content_id` | string/null | Synthetic content identifier when content is shown or played. |
| `content_category` | string/null | Synthetic content category. |
| `watch_seconds` | integer | Watch time attached to play/completion events. Non-video events use `0`. |

## Injected Incident

- `v4.2` begins on `2025-11-17`.
- `v4.2` is served only to `IN-North` and `IN-South`.
- Other regions remain on `v4.1`.
- The affected post-rollout sessions have higher bounce probability, lower completion probability, and shorter watch duration.

## Why Synthetic Data Is Used

The goal is to demonstrate a product analytics workflow without using private production logs. The data intentionally includes a known incident mechanism so the RCA can be evaluated for correctness and reproducibility.

## Limitations

- The incident is intentionally cleaner than most real production incidents.
- The data does not include marketing campaigns, content supply changes, paid acquisition, network quality, or logging outages.
- The analysis supports rollout-related RCA evidence, not randomized causal proof.
